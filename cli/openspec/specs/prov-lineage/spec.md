# prov-lineage Specification

## Purpose
Answer "where did this file come from?" (and "what came from this file?") by
traversing the generation/usage edges the provenance recorder already writes —
resolution of a file reference into the shared `(path, hash)` entity space, the
backward and forward walks, and their tree/JSON renderings.
## Requirements

### Requirement: A file reference resolves to file entities in the shared QName space

The prov module SHALL resolve a lineage file reference against the analysis's stored
PROV document's `inflexa:file-*` entities: an exact `inflexa:path` match, an exact
`inflexa:hash` match, or a hash prefix of at least 6 characters that matches exactly
one entity. An exact path carried by SEVERAL entities (the same path re-written
across runs) SHALL resolve to all of them — each is a genuinely distinct entity and
each is walked and labeled by its hash. An ambiguous hash prefix SHALL fail listing
the candidate hashes and paths; an unmatched reference SHALL fail listing a sample
of the document's known file paths.

#### Scenario: A path shared by two runs resolves to both entities

- **WHEN** `runs/run-001/step-de/output/results.csv` was written in run-001 and re-written with different bytes in run-002, and lineage is asked for that path
- **THEN** both file entities resolve, and the output renders one lineage per entity, each labeled with its content hash

#### Scenario: A unique hash prefix resolves; an ambiguous one fails with candidates

- **WHEN** lineage is asked for a 8-character prefix matching exactly one entity's hash
- **THEN** that entity resolves; and when the prefix matches two entities, the command fails listing both `(path, hash)` candidates instead of guessing

#### Scenario: An unknown reference fails helpfully

- **WHEN** lineage is asked for a path/hash present nowhere in the document
- **THEN** the command fails and lists a sample of the file paths the document does contain

### Requirement: Backward lineage walks generation and usage edges transitively

The traversal SHALL walk backward from each resolved file entity: the entity's
`wasGeneratedBy` activity (the command generation authority, or the step for leaf
files), then that activity's `used` entities, recursing into each. Activity nodes
SHALL carry the context a reader needs to trust the answer: the command line and
exit code for `inflexa:Command`, the tool name for `inflexa:FileToolWrite`, and the
owning step and run (via `wasInformedBy` for commands; via the step activity's own
`inflexa:runId`/`inflexa:stepId` attributes otherwise). Cross-run chains SHALL fall
out of the shared `(path, hash)` entity space with no special handling: a
`source: "prior"` read's entity IS the producing run's entity. An entity with no
recorded generation (staged data, cross-analysis reads) SHALL render as a terminal
input, and an activity with no recorded inputs SHALL render as "no recorded inputs"
— never as an implicit claim that no inputs existed.

Edges that attach through a STEP activity are STEP-GRAIN: a step's step-level reads
and its leaf-file generations connect through step membership — an honest upper
bound, not the per-file fact a command edge states. Renderings SHALL label
step-grain connections distinctly (the tree marks the activity `(step-grain)`), and
a step's empty-side claim SHALL be scoped to its own grain (e.g. "no step-grain
outputs") — command-attributed facts are never asserted absent by a step node.

#### Scenario: An intra-step chain walks A → file → B

- **WHEN** command A wrote `de_results.csv`, command B read it (`source: "step"`) and wrote `heatmap.png`, and lineage is asked for `heatmap.png`
- **THEN** the walk shows heatmap generated by B, B's inputs include `de_results.csv`, and recursing shows it generated by A with A's own inputs beneath

#### Scenario: A cross-run chain walks through the shared entity

- **WHEN** run-002's step read run-001's output (`source: "prior"`) and produced a model file, and lineage is asked for the model file
- **THEN** the walk crosses into run-001: the prior file's node shows run-001's producing command and its inputs

#### Scenario: A leaf file terminates at its step, with step-grain inputs labeled

- **WHEN** a file has only the step-level generation edge (no producing command was recorded) and its step recorded step-level reads
- **THEN** its node shows the step as generator, labeled with the step and run ids and marked step-grain, and the step's step-level reads render beneath it under that marking — never presented as per-file facts

### Requirement: Forward lineage is the reversed walk

With `--forward`, the traversal SHALL walk from each resolved entity to the
activities that `used` it, then to the entities those activities generated,
recursing — answering "what was derived from this file?".

#### Scenario: Forward from a staged input reaches the outputs

- **WHEN** `data/inputs/counts.csv` was read by command A which wrote `de_results.csv`, later read by command B which wrote `heatmap.png`, and forward lineage is asked for `counts.csv`
- **THEN** the walk shows A among its readers, `de_results.csv` beneath A, B beneath that, and `heatmap.png` at the leaf

#### Scenario: A step-level reader surfaces only step-grain outputs

- **WHEN** the step-level registry also recorded the read (the deliberate step-scope redundancy) and forward lineage is asked for the input
- **THEN** the step appears among the readers marked step-grain, carrying only the step's own (leaf) generations — command outputs stay attributed to their commands — and a step with no leaf outputs renders "no step-grain outputs", never a global absence claim

### Requirement: The walk marks cycles and depth cutoffs instead of hiding them

The traversal SHALL keep a visited set and render a revisited entity as an explicit
reference marker rather than re-expanding it (a command that writes and re-reads one
path records `generates ∧ uses` on the same entity — a real cycle). A `--depth`
bound SHALL render an explicit truncation marker at the cutoff, and "unbounded"
SHALL be backed by a hard safety ceiling (1000 hops) that truncates with the same
marker rather than exhausting the call stack on a pathological chain. Nothing in
either rendering SHALL make a truncated or cyclic branch look like a completed one.

#### Scenario: A self-read cycle renders once

- **WHEN** one command both generated and used the same `(path, hash)` entity and lineage is asked for that path
- **THEN** the walk renders the entity once and marks the re-encounter as a reference, and the command terminates

#### Scenario: Depth bound is explicit

- **WHEN** lineage is asked with `--depth 1` on a chain three hops deep
- **THEN** the output ends the bounded branch with a truncation marker, not a clean leaf

### Requirement: Tree and JSON renderings expose the same walk

The command SHALL render `tree` (default) — one lineage per resolved entity, inputs
indented beneath their producing activity — and `json`: a flat graph
`{ roots, nodes, edges }` with nodes keyed by QName carrying kind-specific fields
(path/hash for files; command/exitCode, tool, runId/stepId for activities) and edges
in PROV semantics (`wasGeneratedBy` entity→activity, `used` activity→entity)
regardless of walk direction. No `dot` format SHALL be emitted (descoped
deliberately — no new dependencies, and no dependency-shaped feature either).

#### Scenario: JSON is a flat, direction-independent graph

- **WHEN** the same file's lineage is rendered as `json` backward and forward
- **THEN** both emit `nodes` keyed by QName and `edges` in PROV semantics, differing only in `roots` and reachable subset — a script can re-derive either direction from the edges
