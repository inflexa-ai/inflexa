# record-command-lineage

> **Sequencing**: follows `deepen-run-provenance` (archived 2026-07-06; the `prov-run-events` / `prov-harness-bridge` specs this change modifies are synced in `openspec/specs/`). Informally "change D3" of the harness-integration graph. Every seam fact below was verified against the current tree and the Cortex reference implementation on 2026-07-06.

## Why

The signed provenance document records *that* a step produced each file but not *which command* produced it, nor which reads fed which outputs — the bridge keeps only `producer.type` from the collector's records (`src/modules/harness/prov_bridge.ts:55`) and discards the command string, args, exit code, duration, script path, and the per-command input scoping the sensor already captured and content-attested. Intra-step lineage chains are unrecoverable: when command A writes `de_results.csv` and command B reads it to write `heatmap.png`, the document can only say the step used and generated all of them. This was the deliberate old-Q1 cutover collapse ("bare now, rich later — additive"); the reference implementation ledgers at command granularity — Cortex groups outputs by `Producer` object identity into one activity per command execution, and Nexus draws `used`/generation edges per activity. "Which command, from which inputs, produced this file" is exactly the question a signed provenance document exists to answer.

## What Changes

- **New bus event `prov.command_executed`** — one per surviving producer group (the collector is last-write-wins per output path, so groups are the post-collapse command/file-tool executions). The payload is a discriminated `ProvCommandRef`: a `command` variant carrying `{command, args?, exitCode, durationMs?, scriptPath?, outputs: {path,hash}[], inputs: {path,hash,source}[]}` and a `file_tool` variant carrying `{tool, outputs}` — mirroring Cortex's two activity types. `Producer.timestamp` is deliberately excluded (re-minted on every DBOS replay, `harness/src/provenance/collector.ts:295` — it must never reach QNames or formal positions).
- **New builder `appendCommandExecuted`**: a PROV **activity** `inflexa:cmd-{runId}-{stepId}-{digest(sorted output (path,hash) pairs)}` (`prov:type` `inflexa:Command` / `inflexa:FileToolWrite`) with the execution facts as `inflexa:*` attributes, `wasInformedBy` the step activity, `wasAssociatedWith` the agent, a command-scoped `used` edge per input (including the script via `scriptPath` — Cortex's `scriptIndex` analogue), and `wasGeneratedBy(file, command)` for each output. All relation identifiers deterministic per the existing endpoint-tuple scheme; no formal times.
- **File generation moves from step to command** for produced files: `appendFileWritten` no longer writes `wasGeneratedBy(file, step)` when the file has a producing command activity; **leaf files** (no producer record — e.g. inotify-only observations) keep the step-level generation as fallback (Cortex registers those as leaf entities with no activity; a PROV entity without a generation would be legal but less useful locally).
- **Step-level `prov.input_used` stays unchanged** — it remains the step's attested-inputs registry; the command activities add finer-grained `used` edges onto the *same* `(path,hash)` entities. Redundancy is deliberate and cheap (see design D3).
- **Bridge rework in `createBusArtifactRegistry`**: group `collector.getRecords()` by producer object reference (mirroring the managed reference's grouping), emit one `prov.command_executed` per group before its `prov.file_written` events.
- **Deliberate scope cut**: verification is unit/recorder-level only — no live E2E. The bus→builder→flush→sign pipeline was live-proven by `deepen-run-provenance` days ago; this change adds one event flowing through unchanged machinery.

## Capabilities

### New Capabilities

*(none)*

### Modified Capabilities

- `prov-run-events`: new event + domain types, the `appendCommandExecuted` builder and its record model, the command-QName determinism rule (output-set digest, never producer timestamps), and the generation-edge move (command-produced files generated by the command activity; leaf files by the step).
- `prov-harness-bridge`: the adapter's registration translation gains the producer-grouping stage and the `prov.command_executed` emissions; file/input event mechanics otherwise unchanged.

## Impact

- **cli only** — the `ArtifactRegistry` seam already delivers full `ProvenanceRecord`s (`collector.getRecords()`, `harness/src/provenance/collector.ts:284-326`): `src/types/prov.ts` (+`ProvCommandRef`), `src/types/events.ts` + `src/lib/bus.ts` (event + telemetry), `src/modules/prov/document.ts` (+builder, generation-edge change), `src/modules/prov/prov.ts` (recorder case), `src/modules/harness/prov_bridge.ts` (grouping + emission), tests throughout.
- **harness**: zero changes. **tsprov**: zero changes.
- **Constraints honored**: replay idempotency via deterministic QNames/relation ids and timestamp exclusion; prov-must-sign unchanged; events keyed by `analysisId` with silent unknown-row drop, unchanged.
