## Context

The TUI is Solid (`solid-js`) rendered to the terminal via `@opentui/solid`; Solid components run once and reactivity lives in signals/stores. All colors come from a single module, `src/tui/theme.ts`, which today exports a static `as const` object of 12 semantic roles (Tokyo Night). It is imported directly by exactly two files — `src/tui/app.tsx` (the chat TUI, ~15 reads) and `src/cli/config.tsx` (the settings TUI, ~12 reads) — and consumed as `theme.X`. Because the object is static, color reads are not reactive: nothing repaints if the object is mutated.

Repo constraints that shape this design: no new dependencies (`zod` is already present and is the established boundary-validation tool); `src/lib/env.ts` is the only `process.env` reader; named exports only; domain types instead of raw strings; lib-layer fallible writes return neverthrow `Result`s; keep single-caller helpers in their owning file. Persistence already exists: `src/lib/config.ts` reads/writes `config.json` (currently `{ telemetry: boolean }`), validated with a zod schema, `readConfig()` failing closed and `writeConfig()` returning a `Result`.

## Goals / Non-Goals

**Goals:**

- A reactive theme layer: switching the active theme repaints the live TUI with no restart, driven entirely from Solid signals (matching the renderer's reactivity model).
- An expanded but grounded semantic token vocabulary that keeps every existing token name (no rename churn) and adds tokens for surfaces that already exist or are trivially reachable — including a `syntax` group so markdown/code blocks are themed (today they use OpenTUI's default `SyntaxStyle`).
- ~5 curated dark built-in themes, with `tokyo-night` byte-for-byte identical to today's palette so the default experience is unchanged.
- A theme picker inside the existing `inf config` TUI with live preview, persist-on-save, and revert-on-discard, reusing the screen's draft/saved/dirty machinery.
- Backward-compatible persistence: adding the `theme` key must never drop a user's existing settings (e.g. telemetry consent) on read.
- Zero new dependencies.

**Non-Goals:**

- User-authored theme files loaded from disk, and base16/base24 import. (Both were explicitly deferred; the token model and registry are shaped so a JSON loader could be added later without reworking consumers.)
- Light/dark auto-switching via `renderer.themeMode`. All built-ins are dark; a `variant` field is reserved on the theme type but no auto-flip is built.
- Terminal capability detection / truecolor downsampling / `NO_COLOR` handling. We rely on OpenTUI's native renderer to downsample to 256/16 colors; built-in palettes are authored to stay legible.
- ANSI-index or `"none"`/terminal-default token values. Built-in token values are literal hex; adaptive color intents are a future extension.
- A separate `inf theme` command or a live keybind in the chat TUI — selection lives only in `inf config`.

## Decisions

**1. Reactive global signal in `theme.ts`, not a Solid Context provider.** Replace the static export with a registry plus a module-level `createSignal` holding the active theme id, an accessor `theme()` that returns the active theme's flat colors (so consumers change `theme.X` → `theme().X` — a mechanical one-token edit), and `setTheme(id)`. Rationale: the theme is genuine app-wide singleton state, and there are two independent render roots (`launchTui`, `launchConfig`) — a Context provider would have to be threaded into each, whereas a global signal is read identically from anywhere and matches the existing "import `theme` from `theme.ts`" pattern. Solid signals work fine outside a component tree. The built-in `themes` registry and its type shapes (`Theme`/`ThemeColors`/`ThemeSyntax`) live in a dependency-light `src/lib/themes.ts` (no solid-js, no renderer); `theme.ts` imports the data and layers the reactive accessor over it. Alternatives considered: Context + `useTheme()` (idiomatic in component trees but adds provider plumbing across two roots for no benefit here); keeping a static object and re-rendering manually (defeats the renderer's reactivity, would require imperative re-application to every node — the exact verbosity OpenTUI's binding exists to avoid).

**2. Expanded vocabulary is additive; `theme()` stays flat.** The active accessor returns a flat `ThemeColors` object so existing reads keep their shape (`theme().accent`, `theme().bg`). The registry entry is richer — `{ id, name, variant, colors, syntax }` — but only `colors` is what components read for box/text props. New color tokens: `bgPanel` (elevated chrome: header/status bars), `borderActive` (focused input border), `secondary` (secondary accent), `info` (informational notices — `inf config` currently borrows `success` for its "info" notice, which this fixes). All 12 current names are retained with identical meaning. Rationale: keeps the migration to a `theme.` → `theme().` find-replace, honors "don't rewrite working call sites," and only adds tokens with a real rendered home. Alternative considered: a full OpenCode-style ~50-token set (diff groups, dozen markdown tokens) — rejected as speculative; this app renders no diff view, and the markdown component takes a single `SyntaxStyle`, not per-element tokens. `accent` stays the primary accent (no rename — keeps the 27 call sites stable) and pairs with the new `secondary`; `diff` tokens are intentionally excluded since no diff surface exists (see Non-Goals).

**3. Markdown/code styling derives from a `syntax` token group, built centrally and cached per theme.** `app.tsx` originally held a static `SyntaxStyle.create()` at module scope. Instead, `theme.ts` exposes a shared `syntaxStyle()` accessor that reads the active-theme signal (so markdown recolors on switch) and lazily builds `SyntaxStyle.fromStyles({ ...themes[id].syntax })`, caching the result per `ThemeId`. Because the built-in palettes are immutable, each theme's native style is constructed at most once (total ≤ #themes) and never freed during the process — so there is no disposal/use-after-free concern, and no always-hot memo to rebuild native handles in processes that render no markdown (e.g. `inf config`, which previews themes live but has no `<markdown>`, so the lazy accessor is never called there). Rejected alternatives: a module-scope `createRoot`+`createMemo` (eager and hot, so it rebuilds a native style on every config-preview keypress for no consumer); a double-buffered retirement queue to defer disposal (only worth it if themes become mutable/hot-reloaded — the cache key would then need a version). The `syntax` group is keyed by the token names `@opentui/core`'s `SyntaxStyle` actually accepts (`keyword`, `string`, `comment`, `number`, `function`, `type`, `variable`, `operator`, `punctuation`, …); each entry is `{ fg, bold?, italic? }`. The accepted key set and the `fromStyles` signature MUST be verified against the installed `@opentui/core@0.4.0` — the type is the source of truth, not this list.

**4. `ThemeId` is a domain type; the config schema evolves backward-compatibly.** `ThemeId` is a string-literal union of the built-in ids. It, the `themeIds` tuple, and `DEFAULT_THEME_ID` live in a tiny **solid-js-free** module `theme_ids.ts`, imported by both `theme.ts` and `config.ts`. `config.ts` adds `theme: z.enum(themeIds).catch(DEFAULT_THEME_ID).default(DEFAULT_THEME_ID)` to `configSchema`; `Config` follows via `z.infer`. The `.default` handles configs written before this change (no `theme` key → default), and `.catch` handles an unknown/garbage id on disk (→ default) — **both are load-bearing**: making `theme` a plain required field would make `safeParse` reject any pre-existing `config.json`, and since `readConfig()` falls back to `{ telemetry: false }` on parse failure, that would silently wipe a user's telemetry consent on the next read. The ids living in their own module is also load-bearing for layering: `src/index.ts` reads config at startup for **every** command, so if `config.ts` imported the full `theme.ts` it would drag solid-js + the reactive registry onto non-TUI paths (`inf login`, `whoami`, …). Import direction is strictly `config.ts → theme_ids.ts` and `theme.ts → theme_ids.ts`; `theme.ts` MUST NOT import `config.ts` (see Decision 5) to avoid a cycle.

**5. Active theme initialized from persisted config at launch, keeping `theme.ts` config-free.** Each `launch*` function (`launchTui`, `launchConfig`) calls `setTheme(readConfig().theme)` before `void render(...)`, where it already resolves data first. The signal's initial value is `DEFAULT_THEME_ID`, so a render before init is still valid. Rationale: if `theme.ts` read config itself it would import `config.ts`, which imports the ids module for the `theme` enum — and a back-import would form a cycle. Two explicit init call sites are cheap and keep the dependency arrow one-directional. Alternative considered: a single `initTheme()` in `theme.ts` (rejected — reintroduces the cycle or forces a lazy/function-level import that obscures the data flow).

**6. The `inf config` picker is a visible theme list with preview-on-highlight, reusing the draft/saved model.** `theme` joins the screen's draft state alongside `telemetry`. The built-in themes render as a vertical list of named rows (the draft selection marked radio-style), folded into the screen's existing up/down row navigation. Moving the highlight onto a theme row calls `setTheme(id)` for instant live preview **and** sets `draft.theme = id` — preview and selection are unified, the expected behavior for a theme chooser: every option is visible and previewed as you arrow through it. `dirty()` extends to compare `draft.theme` against `saved.theme`; `save()` persists `{ ...draft }`; the quit/discard path calls `setTheme(saved().theme)` so abandoning unsaved changes restores the previously-saved theme. Up/down moves across all rows; space/enter still toggles the `telemetry` boolean. Rationale for a list over an inline `‹ name ›` cycle: with ~5 themes, showing every option at once and previewing on highlight is more discoverable and faster to compare than blind cycling. The chat TUI (`app.tsx`) exposes no in-session switch control; it applies the persisted theme at launch only (see Decision 5).

**7. Module-level precomputed color constants must become reactive reads.** In Solid, a color is only reactive if it is read inside a tracking scope (JSX, `createMemo`, effect). `config.tsx` currently computes `const noticeColors: Record<...> = { info: theme.success, ... }` once at import — frozen to whatever theme was active at module load. This (and any similar precomputed map) MUST be rewritten as a function/inline read of `theme()` so it tracks the active theme. This is the single most likely correctness trap in the migration and is called out in Risks.

## Risks / Trade-offs

- [Precomputed color maps don't react] → Decision 7: audit `theme.ts` consumers for any color value computed outside a tracking scope and inline it; verify by switching theme at runtime and confirming every surface (including config notices and code blocks) recolors.
- [Adding `theme` to the config schema regresses existing users' settings] → Decision 4: `.default` + `.catch` keep `safeParse` succeeding for old/garbage configs; verify with a `config.json` that has `telemetry` but no `theme` (telemetry preserved) and one with `theme: "bogus"` (falls back to default, telemetry preserved).
- [`SyntaxStyle.fromStyles` shape or accepted token keys differ in `@opentui/core@0.4.0`] → Decision 3: verify the API against the installed version before authoring the `syntax` group; treat the type as source of truth.
- [Import cycle `theme.ts ↔ config.ts`] → Decisions 4–5 fix the arrow at `config.ts → theme.ts` and move config reads into `launch*`; `bun run typecheck` and a smoke run confirm no cycle.
- [Built-in palettes illegible after OpenTUI downsamples on a 256/16-color terminal] → Non-goal to detect capability, but author each built-in from its canonical, contrast-checked palette and spot-check at least one in a non-truecolor terminal (`COLORTERM` unset).
- [Live preview left applied after the user discards] → Decision 6 reverts to `saved().theme` on the discard path.

## Migration Plan

Single in-repo change, no data migration. `tokyo-night` is byte-for-byte the current palette, so users who never open the picker see no change. The config schema change is forward- and backward-compatible (Decision 4): old binaries ignore the unknown `theme` key (zod strips unknown keys on read); new binaries default it. No rollback steps beyond reverting the change.

## Open Questions

- None blocking. Exact `syntax` token keys and the `SyntaxStyle.fromStyles` signature are resolved during implementation against `@opentui/core@0.4.0` (Decision 3); the precise built-in palette hex values are an implementation detail authored from canonical sources.
