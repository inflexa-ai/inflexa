## Context

Local embeddings already run through a pinned `llama-server` sidecar serving a GGUF over loopback, realizing the harness `EmbeddingProvider` seam; `embedding.mode` selects between it, a direct api-key endpoint, and off. Two prior changes reshaped the ground this one builds on: the model became a build-time embedded asset (the compiled binary materializes it offline), and `embedding.dimensions` was added to the config schema so the harness can size each per-analysis search index to the provider's advertised width. But the surfaces around the provider still assumed one model at one width. `promptEmbeddingMode` offered a single "local" choice whose copy said "downloaded once" — wrong in the binary — and had no path for a GGUF the user already owns. `createLocalEmbeddingProvider` hardcoded `dimensions: 384`. `ensureEmbedderReady` checked `env.embeddingModelPath` (the built-in location) unconditionally. `inflexa config` rendered `embedding.*` as flat per-field rows, which cannot honestly present a mode-discriminated union. And there was no way to discover which models a user-supplied api-key endpoint actually serves — the proxy's model discovery is chat-only and hardwired to the local proxy.

## Goals / Non-Goals

**Goals:**

- Let the user choose among the built-in model, their own local GGUF, an api-key endpoint, and off — with copy that matches the install context.
- Support a local model of any width: one measured value drives both the sidecar request and the advertised width the harness sizes each index to, so the two can never disagree.
- Stop the launch gate from failing on the built-in location when a custom model is configured elsewhere.
- Make `inflexa config` present embeddings as what they are — one active backend — and let the user reconfigure it through the existing dialog components without hand-editing JSON.
- Let api-key configuration pick from the endpoint's real model ids instead of a typed-from-memory string.

**Non-Goals:**

- No automatic re-embedding when the backend (and therefore the vector width) changes: each analysis's search index keeps the width it was created at, and re-profiling is the user's action. The loud switch-backend warning is the mitigation.
- No fix for pointing an EXISTING analysis at a different width — that still fails at the pgvector upsert (a per-analysis index keeps its creation width). Tracked in #164; this change makes it reachable from more configurations but does not address it.
- No sidecar spawn from `inflexa config`: the settings screen stays pure config editing. Only `inflexa setup` probes a model.
- No new config surface: `embedding.dimensions`/`modelPath`/`apiKey`/`baseURL`/`model` already exist; this change only populates them from more paths.

## Decisions

1. **A four-way backend picker replaces the three-way mode picker.** The former single "local" option conflated two very different things — a bundled model that needs nothing versus pointing at a GGUF you already have — and its copy implied a download even where the model is an embedded asset. Split "local" into BUILT-IN and YOUR OWN GGUF, keeping api-key and off; picker copy is install-context-aware (no "download" claim in a compiled binary). `--embeddings local` maps to the built-in model only. *Rejected: a `--embeddings custom=/path` flag form.* A custom path is inherently interactive (the file must exist, its location is per-user), and a mistyped non-interactive path would silently misconfigure; custom stays interactive-only, so the flag has no custom form.

2. **One width, measured at setup, drives both the sidecar request and the advertised `provider.dimensions`.** `createLocalEmbeddingProvider` takes `deps.dimensions` and uses that single value for the sidecar's `dimensions` request param AND the advertised `provider.dimensions`, defaulting to `LOCAL_EMBEDDING_DIMENSIONS` (384) when unset. Because the harness sizes each index to the advertised width, a request/advertised split would silently corrupt indexes — one value makes disagreement impossible. Setup measures a custom model's width through the sidecar probe: `verifyModel` asserts 384 for the built-in (SHA-256-pinned, so any other width is a corrupt asset) and omits the assertion for a custom GGUF, returning whatever positive width it emits. That width is recorded as `embedding.dimensions` **only when it differs from 384**, keeping config minimal — a 384-dim custom model persists just its path and lets the provider default apply. *Rejected: asserting a fixed width for custom models.* A user's model width is unknown ahead of time; measuring is the only correct behavior, and a zero-width probe is still `verify_failed` (an unusable model).

3. **Width is measured at setup but ENTERED in `inflexa config`.** `inflexa config` deliberately never spawns the sidecar — materializing the runtime and probing a model (with its one-time ~10s macOS binary scan) belongs to `inflexa setup`, not a settings screen. So the config flow cannot measure a custom GGUF's width; it collects the width as a typed value, and a mistyped width is deliberately NOT guarded here (issue #164: a width disagreeing with an analysis's existing index fails at the pgvector upsert). Config edits write `config.json` only — no acquire, download, or verify — exactly like a hand-edited config; correctness is re-established at the next run by the readiness gate and the profile dimension probe. *Rejected: having config spawn the sidecar to measure.* It would drag the whole runtime-materialization-and-probe path into an otherwise pure config editor; setup already owns that path, and `config` is the "I know what I'm doing" surface.

4. **`api_models` is its own module, not a parameter bolted onto `modules/proxy/models.ts`.** The api-key backend lists what its user-supplied endpoint actually serves so the user selects a real id instead of typing one from memory (a typo surfaces only as a failed embed, late). `proxy/models.ts` lists the local CLIProxyAPI's CHAT models and is hardwired to `env.cliproxyApiUrl`; the api-key embedding backend talks DIRECTLY to a user-supplied endpoint — never through the chat proxy, which serves no embeddings route. Same wire shape, different host, credential, and domain (embedding-capable ids, not chat), so it is its own fetch. The `/models` payload carries no "this is an embedding model" flag, so ids are filtered by an `/embed/i` heuristic; an over-narrow match is safe by construction because a no-match listing reports `no_models` and the caller falls back to free-text entry, so an unconventionally named model stays reachable. A credential-leak guard runs before any fetch: it refuses to send the `Authorization: Bearer` key over cleartext `http://` to a non-loopback host (loopback is exempt for local inference servers). *Rejected: adding `baseURL`/`token` params to the proxy's discovery.* It would couple the local chat proxy to a remote embedding endpoint and blur the "never through the chat proxy" boundary.

5. **The readiness gate checks the CONFIGURED model path, not the built-in location.** `ensureEmbedderReady` reads `embedding.modelPath`, falling back to `env.embeddingModelPath` only when none is recorded (a legacy/hand-edited config). A custom GGUF lives at the user's own path, so checking the built-in location would spuriously fail the launch gate for a file the run was never going to use. The gate still never spawns the sidecar or re-verifies — beyond one-time runtime materialization (self-healing, like the launch gate's container provisioning), existence checks are all the hot path pays; a missing model directs the user to `inflexa setup` or to point `embedding.modelPath` at their GGUF.

6. **The `inflexa config` embedding UI is one summary row plus a chained dialog flow over existing components.** `embedding.*` is a mode-discriminated union, so a single summary row naming the active backend and its distinguishing detail is the honest rendering — the fields of an inactive backend are noise. Activating the row opens a `SelectDialog` backend picker; the chosen backend drives per-backend follow-ups built from the existing `SelectDialog` / `PromptDialog` / `FilePicker` (a custom GGUF: file picker then a width prompt; api-key: key, base URL, a fetched model selection or free-text fallback, then a width), never a bespoke form. Each step closes itself before pushing the next (the stack never grows), and the draft's embedding block is replaced **wholesale** only at a branch's final step. So cancelling any step is a clean abort — there is no partially-applied backend to unwind — and switching backends never inherits a stale field (the api key from a former api-key config never rides along into `local`). The api key is never printed on the summary row (a remote secret), though its own edit prompt may show it.

## Risks / Trade-offs

- [An entered-not-measured width in `inflexa config` can be wrong] → deliberately unguarded; the readiness gate and the profile dimension probe catch a genuinely broken config at the next run, and a width that merely disagrees with an existing analysis's index is the known #164 limitation, out of scope here.
- [Switching backends strands existing analyses' indexes at the old width] → unchanged behavior, surfaced by the loud switch-backend warning; automatic re-embedding stays a deliberate non-feature.
- [The `/embed/i` model-id heuristic can miss an unconventionally named embedding model] → safe by construction: a no-match listing degrades to free-text entry, so the endpoint is still configurable — the failure mode is "type the id yourself", never "cannot configure".
- [A custom GGUF is trusted on the user's word for its width in `config`, and only probed in `setup`] → accepted: `setup` measures and records the real width; `config` is the lower-ceremony surface where the user owns the value they type.

## Migration Plan

Code-only; the config schema is unchanged. Existing `embedding.mode = "local"` configs keep working: the built-in `modelPath` is unchanged, and a config with no `dimensions` defaults to 384 exactly as before. No data migration and no prod-visible behavior change beyond the setup/config copy and the added choices. Rollback = revert.

## Open Questions

None. The width split (setup measures, config enters), the four-way picker, and the api-key model discovery were decided with the user; the cross-width existing-index failure is tracked in #164 and explicitly out of scope.
